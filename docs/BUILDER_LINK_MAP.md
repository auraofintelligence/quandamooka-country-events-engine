# Builder Link Map

This repo is designed to sit beside the existing Markdown builder ecosystem rather than replace it.

## New builders in this repo

- `event-brief.md` - event intention, audience, place, purpose, public needs and private boundaries.
- `event-resource.md` - venue, supplier, artist, gear, planner, transport or support profile.
- `simulation-brief.md` - attendance, movement, site load, environment, safety, approval gaps and decision.
- `aftercare-report.md` - cleanup, gratitude, incidents, evidence, visitor pressure and next-time changes.
- `place-use-record.md` - place purpose, carrying limits, permission path, site requirements and private boundaries.
- `approval-trail.md` - responsible body, source link, status, conditions and next check.
- `supplier-request.md` - quote or availability request with clear public/private boundary.
- `event-run-sheet.md` - bump-in, program flow, critical checks, escalation and pack-down.
- `crew-roster.md` - roles, shifts, training, contact boundary and roster gaps.
- `risk-incident-log.md` - risk signals, controls, incident channel, public statement and aftercare follow-up.
- `value-evidence.md` - benefit claims, evidence, who benefits, who carries load and next measures.
- `source-trail.md` - source title, checked date, confidence, uncertainty, update trigger and public boundary.
- `market-intake.md` - stallholder, food, artist, host-space or market add-on intake.
- `sim-crowd-flow.md`, `sim-transport.md`, `sim-logistics.md`, `sim-environment.md`, `sim-vibe-access.md`, `sim-guardian.md`, `sim-local-economy.md` - specialist simulation records for agent lanes.
- `sovereign-pipeline-alignment.md` - pipeline alignment record for layer path, builder inputs, agent roles, tool stack, freshness and public boundary.
- `public-event-notice.md`, `change-update-notice.md`, `transport-access-notice.md`, `aftercare-summary-notice.md` - public-safe notice records.

## Suggested file flow

1. Create `event-brief.md`.
2. Create or link relevant `business-profile.md`, `public_noticeboard.md`, `asset-list.md`, `shared-assets.md` and `equipment-wishlist.md`.
3. Create the relevant run files: place use, approval trail, supplier request, run sheet, crew roster, risk log, value evidence or source trail.
4. Create `simulation-brief.md` and specialist simulation files when scale or risk requires them.
5. Publish only approved public notices.
6. Export `aftercare-report.md` after the event.
7. Feed the aftercare summary back into the calendar, noticeboard and next simulation pass.
